# Web Development Glossary

Standardized technical terms for web development. Each entry covers the precise technical term, its common alias, a system-level definition for coding agents, and plain-language guidance for human readers — beginner to advanced.

## Accessibility

| Preferred Technical Term | Standard / Widely Accepted Term | Description (For Systems) | Guidance (For Humans) |
|---|---|---|---|
| `aria-label` | Tooltip label | HTML attribute providing a text alternative for an element, read aloud by screen readers when the element receives focus. No visible rendering. | "A short description attached to a button or icon that screen readers announce aloud. Invisible to sighted users unless surfaced via tooltip." *Used when no visible label text exists — e.g., icon-only buttons.* |
| `aria-describedby` | Popover reference | HTML attribute linking an element to one or more elements containing extended descriptive content; exposed to assistive technologies as supplemental description. | "Connects an input or control to a longer explanation elsewhere on the page. Screen readers read both the label and this description." *Value is the `id` of the describing element — supports multiple ids space-separated.* |
| `aria-live` | Live region | HTML attribute designating a region that will receive dynamic content updates, instructing assistive technologies to announce changes automatically. Values: `polite`, `assertive`, `off`. | "Tells screen readers to automatically announce changes to a section — like status messages or notifications — without the user needing to navigate there." *`assertive` interrupts immediately; `polite` waits for a pause in speech.* |
| `aria-expanded` | Open/close state | Boolean ARIA attribute communicating the expanded or collapsed state of a control (e.g., accordion, dropdown) to assistive technologies. | "Signals to screen readers whether a toggle, menu, or accordion section is currently open or closed." *Must be updated in JS whenever the controlled element's visibility changes.* |
| `role` attribute | Element role override | ARIA attribute that explicitly defines the semantic role of an element for assistive technologies, overriding or supplementing native HTML semantics. | "Tells browsers and screen readers what an element is — for example, marking a `<div>` as a `button` or `dialog`." *Prefer native HTML elements over role overrides when possible.* |
| `focus management` | Keyboard trap / focus control | Programmatic control of which element holds keyboard focus — especially critical in modals, dialogs, and SPAs — to maintain logical, accessible navigation order. | "Controls where keyboard focus goes when something opens or closes on the page, so keyboard-only users aren't lost." *Modals must trap focus inside themselves; focus must return to the trigger element on close.* |
| `skip navigation link` | Skip to content | A visually hidden anchor link rendered first in the DOM, allowing keyboard and screen reader users to bypass repetitive navigation and jump directly to main content. | "A hidden link at the top of the page that lets keyboard users skip the nav menu and jump straight to the main content." *Must be focusable and visible when focused — often the first tab stop on the page.* |

## UI Patterns & Components

| Preferred Technical Term | Standard / Widely Accepted Term | Description (For Systems) | Guidance (For Humans) |
|---|---|---|---|
| `modal dialog` | Modal / popup | UI element rendered above the page layer requiring user interaction before returning control to the parent window; typically disables interaction with underlying content via an inert backdrop. | "A box that appears over the page and demands your attention before you can continue. Everything behind it is blocked." *Requires `aria-modal`, `role="dialog"`, focus trap, and `Escape` key dismissal for full accessibility.* |
| `collapsible section` | Accordion item | UI element toggling between expanded and collapsed states using a trigger control, revealing or hiding associated content while maintaining vertical stacking with sibling items. | "Click a heading to show or hide the content under it. Common in FAQs and settings panels." *Each item should use `aria-expanded` and `aria-controls` to expose state to assistive technologies.* |
| `carousel` | Slideshow / slider | UI pattern cycling through a sequence of content panels using transitions (slide, fade, etc.) with optional auto-advance and navigation controls (dots, arrows, swipe). | "A series of images or cards that rotate through one at a time. You can usually swipe or click arrows to move between them." *Auto-advance carousels must pause on hover/focus and expose controls to assistive technologies.* |
| `skeleton screen` | Loading skeleton / placeholder | UI pattern using inert placeholder shapes matching the expected content layout, displayed during data fetching to reduce perceived latency and prevent layout shift. | "The gray outlined shapes you see while a page loads — they show where content will appear so the page doesn't feel empty." *Reduces Cumulative Layout Shift (CLS) versus spinner-only approaches.* |
| `off-canvas navigation` | Hamburger menu / slide-out menu | Navigation pattern where menu content is positioned off-screen and toggled into the viewport via CSS transforms or transitions, typically triggered by a hamburger icon control. | "The side menu that slides in when you tap the icon with three lines. Common on mobile — hides the nav until you need it." *Requires `aria-expanded` on trigger, focus management into the panel, and `Escape` dismissal.* |
| `breadcrumb navigation` | Breadcrumbs | Secondary navigation pattern displaying the user's hierarchical path through site structure as a series of links to ancestor pages, typically separated by a delimiter character. | "The trail of links at the top of a page — like `Home > Products > Shoes` — showing where you are and letting you jump back easily." *Wrap in `<nav aria-label="Breadcrumb">` with `aria-current="page"` on the last item.* |
| `tooltip` | Hover hint | Transient non-interactive UI element displaying contextual information about a target element, triggered by pointer hover or keyboard focus, dismissed on blur or pointer leave. | "The small label that appears when you hover over a button or icon, explaining what it does." *Tooltips must also appear on keyboard focus, not hover alone — use `aria-label` or `aria-describedby` as the text source.* |
| `popover` | Floating panel | A non-modal floating content panel anchored to a trigger element, displaying richer content than a tooltip (forms, menus, previews) and dismissed by interaction outside its boundary. | "A small panel that pops open next to a button, showing more options or details without opening a full modal." *HTML now has a native `popover` attribute — prefer it over custom implementations where browser support allows.* |
| `toast notification` | Snackbar / flash message | Ephemeral non-blocking UI notification rendered at the screen edge (typically bottom or top-right), auto-dismissing after a timeout while preserving user workflow. | "A small message that briefly appears at the corner of the screen — like 'Saved!' — then disappears on its own." *Must be in an `aria-live` region so screen readers announce it; ensure sufficient display time for readability.* |
| `infinite scroll` | Auto-loading content | Content loading pattern that automatically fetches and appends paginated data as the scroll position approaches the viewport bottom, using an Intersection Observer or scroll event threshold. | "More content automatically loads as you scroll down, so you never hit a 'Next Page' button." *Can harm accessibility and performance — always provide a manual load trigger as a fallback.* |
| `virtual list` | Windowing / virtualization | Rendering optimization where only the DOM nodes visible within the current viewport are rendered; off-screen list items are unmounted and remounted as the user scrolls. | "When showing thousands of rows or cards, only the ones you can see are actually in the page — the rest are created on demand as you scroll." *Libraries: `react-window`, `@tanstack/virtual`. Critical for performance in long lists.* |
| `tabs` | Tab bar | UI pattern organizing content into discrete panels activated by sibling tab controls, with one panel visible at a time and tab state tracked via `aria-selected`. | "Clickable labels at the top of a section that switch what content is shown below, like browser tabs." *Follow the ARIA Authoring Practices Guide tab pattern: arrow key navigation between tabs, `aria-controls` linking each tab to its panel.* |
| `drag and drop` | DnD / sortable | Interaction pattern allowing users to reorder, move, or transfer UI elements by pressing, dragging, and releasing via pointer or touch input, with state managed in JavaScript. | "Click and hold an item, move it to a new spot, and release to place it there." *Requires keyboard-accessible alternative interactions; libraries: `@dnd-kit/core`, `react-beautiful-dnd`.* |

## Performance

| Preferred Technical Term | Standard / Widely Accepted Term | Description (For Systems) | Guidance (For Humans) |
|---|---|---|---|
| `lazy loading` | Deferred loading | Performance strategy delaying the fetch and initialization of non-critical resources (images, components, iframes) until they are needed — typically on viewport entry via Intersection Observer. | "Images and components only load when you're about to see them, making the initial page load much faster." *Use `loading="lazy"` on `<img>` natively; use dynamic `import()` for code splitting React components.* |
| `code splitting` | Chunking / dynamic import | Build-time or runtime technique dividing a JavaScript bundle into smaller chunks loaded on demand, reducing initial parse and execution time. | "Instead of loading all the website's code upfront, it loads only what's needed for the current page — the rest loads when you navigate there." *Implemented via `React.lazy()` + `Suspense`, or route-level splitting in Next.js / Vite.* |
| `CDN (Content Delivery Network)` | Edge caching | Distributed network of proxy servers caching static assets at geographically proximate edge nodes, reducing round-trip latency by serving requests from the nearest point of presence. | "Copies of your website's files are stored on servers all over the world, so everyone loads them from a server nearby — faster for everyone." *Static assets (images, fonts, JS/CSS bundles) should always be CDN-served in production.* |
| `debounce` | Input delay / rate limit | Rate-limiting technique deferring function execution until a specified idle period has elapsed since the last invocation, collapsing rapid repeated calls into a single execution. | "When you type in a search box, the site waits until you pause before searching — so it doesn't fire on every single keystroke." *Distinct from throttle: debounce delays until idle; throttle executes at a fixed maximum rate.* |
| `throttle` | Rate limiting | Rate-limiting technique ensuring a function executes at most once per specified time interval regardless of how many times it is invoked within that period. | "Limits how often something can run — like capping a scroll handler to fire only once every 100ms even if the user scrolls rapidly." *Use throttle for continuous events (scroll, resize); use debounce for terminal events (search input, resize end).* |
| `memoization` | Caching computed values | Optimization technique caching the return value of a pure function keyed by its input arguments, skipping recomputation when the same inputs recur. | "The site remembers the result of an expensive calculation so it doesn't repeat the work if nothing has changed." *In React: `useMemo` for values, `useCallback` for functions, `React.memo` for component renders.* |
| `tree shaking` | Dead code elimination | Build-time static analysis removing unused module exports from the final bundle, reducing payload size. Effective only with ES module (`import`/`export`) syntax. | "The build process automatically removes code that's never used anywhere in your app, keeping the final file size smaller." *Requires ES modules — CommonJS (`require`) is not tree-shakeable.* |
| `Core Web Vitals` | Page experience metrics | Google's set of standardized UX performance metrics: LCP (Largest Contentful Paint), INP (Interaction to Next Paint), and CLS (Cumulative Layout Shift), used in search ranking signals. | "Three key scores Google measures to judge if your page is fast and stable: how fast the main content appears, how quickly it responds to clicks, and whether things jump around while loading." *LCP target < 2.5s; INP target < 200ms; CLS target < 0.1.* |
| `critical rendering path` | Above-the-fold loading | The sequence of steps the browser takes to convert HTML, CSS, and JS into pixels on screen; optimization minimizes render-blocking resources to accelerate first paint. | "The chain of work the browser does to show the first visible frame of your page — optimizing it means the page appears faster." *Inline critical CSS, defer non-critical JS, preload key fonts and images.* |
| `preloading / prefetching` | Resource hints | Browser directives (`<link rel="preload">`, `<link rel="prefetch">`) instructing the browser to fetch resources early: `preload` for current-page critical assets, `prefetch` for likely next-navigation resources. | "`preload` tells the browser to download an important file immediately even before it's encountered in the HTML. `prefetch` quietly downloads files the user might need on the next page." *Overuse of `preload` competes with critical resources — use selectively.* |

## Rendering & Architecture

| Preferred Technical Term | Standard / Widely Accepted Term | Description (For Systems) | Guidance (For Humans) |
|---|---|---|---|
| `client-side rendering (CSR)` | Single-page app rendering | Rendering strategy where HTML is minimal on initial load and the DOM is constructed entirely by JavaScript in the browser after JS bundle download and execution. | "The page arrives mostly empty and JavaScript builds everything in your browser. Fast after the first load, but slower to show the first screen." *Weak for SEO without additional hydration strategies.* |
| `server-side rendering (SSR)` | Server rendering | Rendering strategy where full HTML is generated on the server per request and sent to the client, enabling immediate content display before JavaScript hydration. | "The server builds the complete page and sends it ready to view — faster first load, better SEO, but the server does more work per request." *Hydration attaches React event listeners to the server-rendered HTML after it arrives.* |
| `static site generation (SSG)` | Pre-rendering / static rendering | Build-time rendering strategy that generates all HTML pages ahead of time as static files, served directly from a CDN without per-request server computation. | "All pages are built once when you deploy the site and saved as plain HTML files — extremely fast to serve but requires a rebuild for content changes." *Ideal for content-stable pages (docs, marketing). Combine with ISR for dynamic content.* |
| `incremental static regeneration (ISR)` | On-demand revalidation | Hybrid rendering strategy (Next.js-native) that serves statically generated pages while revalidating and regenerating them in the background at defined intervals or on-demand. | "Like static pages, but they quietly refresh in the background on a schedule, so content stays fresh without a full site rebuild." *Configured via `revalidate` in Next.js `fetch` options or page config.* |
| `hydration` | Client activation | Process by which client-side JavaScript attaches event listeners and reactive state to server-rendered or statically generated HTML, making the page interactive. | "After the server sends the HTML, JavaScript 'wakes it up' — attaching all the click handlers and interactivity." *Hydration mismatch errors occur when server and client render different output — a common SSR debugging issue.* |
| `island architecture` | Partial hydration | Rendering pattern where most of the page is static HTML and only discrete interactive components ("islands") are hydrated with JavaScript, minimizing JS execution cost. | "Most of the page is plain, fast HTML. Only the interactive parts — like a search bar or cart widget — run JavaScript." *Used by Astro, Fresh. Reduces Time to Interactive significantly on content-heavy sites.* |
| `client-side routing` | SPA navigation / History API routing | Navigation mechanism in SPAs where view transitions are handled by JavaScript intercepting link clicks and updating the DOM without full page reloads, using the History API or hash fragments. | "Clicking links doesn't reload the page — JavaScript swaps out the content instantly, making navigation feel like a native app." *`pushState` / `replaceState` manage the URL; `popstate` event handles back/forward navigation.* |
| `edge functions` | Edge compute / Edge middleware | Server-side functions deployed and executed at CDN edge nodes geographically near the user rather than in a centralized origin server, minimizing round-trip latency for dynamic logic. | "Server-side code that runs at a location near the user instead of a central data center — faster responses for things like auth checks, redirects, and personalization." *Available via Vercel Edge, Cloudflare Workers, Netlify Edge Functions.* |
| `monorepo` | Single repository / unified codebase | Repository structure containing multiple distinct packages or applications managed together, sharing tooling, dependencies, and CI/CD pipelines under one version-controlled root. | "Multiple projects — like a web app, a mobile app, and a shared component library — all live in the same code repository and share code easily." *Tooling: Turborepo, Nx, pnpm workspaces.* |
| `micro-frontend` | Federated UI / modular frontend | Architectural pattern decomposing a frontend application into independently deployable, team-owned UI fragments composed at runtime or build time into a unified user experience. | "Different teams own and deploy separate pieces of the same website independently — like one team owns the checkout and another owns the product pages." *Implemented via Module Federation (Webpack/Rspack), iframes, or web components.* |

## Data & APIs

| Preferred Technical Term | Standard / Widely Accepted Term | Description (For Systems) | Guidance (For Humans) |
|---|---|---|---|
| `REST API` | Web API / HTTP API | Architectural style for network APIs using standard HTTP methods (GET, POST, PUT, PATCH, DELETE) with resources identified by URLs and data exchanged as JSON or XML. | "A standardized way for your website to request and send data to a server — like asking a server for a list of products and getting back JSON data." *Stateless: each request must contain all context; server holds no session between calls.* |
| `GraphQL` | Graph API / query API | Query language and runtime for APIs enabling clients to request precisely the data fields they need in a single request, with a strongly-typed schema defining available data and operations. | "Instead of the server deciding what data to send, you tell it exactly what you need — no more, no less. One request can fetch data from multiple sources." *Eliminates over-fetching and under-fetching; single endpoint vs multiple REST routes.* |
| `webhook` | HTTP callback / event push | HTTP-based event notification pattern where a server sends an HTTP POST request to a pre-registered client URL when a specified event occurs, inverting the typical request-response model. | "Instead of your site constantly asking 'did anything happen?', the other service automatically sends a message to your site the moment something does." *Receiving server must respond with 2xx quickly; process asynchronously via a queue.* |
| `optimistic UI update` | Optimistic rendering | UX pattern that immediately updates the UI to reflect the expected result of an API mutation before the server response is received, rolling back on failure. | "When you click 'Like', the counter goes up instantly — the site assumes it will work and corrects itself if it fails." *Requires rollback logic on error; increases perceived responsiveness significantly.* |
| `pagination` | Page-by-page loading | Data retrieval pattern dividing large result sets into discrete pages, returned via offset/limit or cursor-based parameters, reducing payload size per request. | "Instead of loading 10,000 results at once, the site loads 20 at a time and gives you buttons to navigate through pages." *Cursor-based pagination (using an ID/timestamp as the next-page token) is more stable than offset pagination for real-time data.* |
| `caching (HTTP)` | Browser cache / response cache | Mechanism for storing HTTP responses (via `Cache-Control`, `ETag`, `Last-Modified` headers) to serve repeated requests from local or intermediary storage, reducing origin load and latency. | "Your browser saves a copy of files it's already downloaded so it doesn't re-download them every time you visit — making repeat visits much faster." *`Cache-Control: max-age=31536000, immutable` for hashed static assets; `no-store` for sensitive data.* |
| `environment variables` | Config vars / env vars | Runtime configuration values injected into an application's process environment (not hardcoded in source), used to manage secrets, API keys, and environment-specific settings across deployments. | "Settings that change between development, testing, and production — like database addresses or API keys — stored outside the code so they're not accidentally shared." *Never commit secrets to version control; use `.env.local` for local dev, secrets managers for production.* |

## HTML & CSS Fundamentals

| Preferred Technical Term | Standard / Widely Accepted Term | Description (For Systems) | Guidance (For Humans) |
|---|---|---|---|
| `semantic HTML` | Semantic elements / meaningful markup | Use of HTML elements that communicate the meaning and structure of content (`<article>`, `<nav>`, `<section>`, `<header>`, `<main>`, `<aside>`) rather than generic layout containers. | "Using the correct HTML tag for what it represents — not just `<div>` for everything — so browsers, search engines, and screen readers understand the page structure." *Semantic HTML improves SEO, accessibility, and maintainability simultaneously.* |
| `CSS custom properties` | CSS variables | User-defined properties in CSS (declared as `--property-name: value`) that store reusable values, scoped to the cascade and modifiable at runtime via JavaScript. | "Variables in CSS that let you define a color or size once and reuse it everywhere — change it in one place and it updates everywhere." *Scoped to the element they're declared on; `:root` scope makes them global. Runtime-modifiable unlike preprocessor variables.* |
| `CSS specificity` | Style priority / override rules | Algorithm browsers use to determine which CSS rule applies to an element when multiple conflicting rules match, calculated by counting selector types (inline > ID > class > element). | "When two CSS rules target the same element with conflicting styles, specificity determines which one wins. Inline styles beat IDs, which beat classes, which beat element selectors." *!important overrides all specificity — avoid it; it creates cascading specificity wars.* |
| `CSS cascade layers` | `@layer` | CSS feature allowing authors to define explicit ordering of style layers (`@layer`), providing deterministic control over specificity and override order independent of selector weight. | "A way to organize CSS into named priority groups so you control which styles win without fighting specificity battles." *`@layer utilities` overrides `@layer base` regardless of selector specificity within each layer.* |
| `CSS containment` | `contain` property | CSS property (`contain: layout`, `paint`, `size`, `style`) instructing the browser that an element's subtree is independent from the rest of the document, enabling rendering optimizations. | "Tells the browser that changes inside an element won't affect the rest of the page, letting it optimize repaints and reflows." *`content-visibility: auto` extends this concept for off-screen rendering skip — major performance win for long pages.* |
| `responsive web design` | Adaptive / fluid design | Design methodology using fluid grids, flexible images, and CSS media queries (and container queries) to ensure optimal rendering across all viewport sizes and device orientations. | "The website automatically rearranges and resizes itself to fit any screen — phone, tablet, or desktop — without needing a separate mobile version." *Modern approach adds container queries (`@container`) for component-level responsiveness independent of viewport.* |
| `CSS container queries` | Component-based responsiveness | CSS feature enabling style rules to respond to the size of a parent container element rather than the viewport, allowing truly self-contained responsive components. | "Instead of a component changing based on screen size, it changes based on how big its own container is — so the same component works in a sidebar or a full-width area." *`@container` syntax; requires `container-type` on the parent.* |
| `form validation` | Input validation | Process of verifying user-provided input against defined rules — client-side for immediate feedback (HTML5 constraint API, JS) and server-side for security enforcement (never trust client validation alone). | "The site checks what you typed is correct — like a valid email address or a required field — before processing it. The check happens in your browser for speed and again on the server for security." *Client-side validation is UX only; server-side validation is mandatory for security.* |

## JavaScript & Browser

| Preferred Technical Term | Standard / Widely Accepted Term | Description (For Systems) | Guidance (For Humans) |
|---|---|---|---|
| `event delegation` | Bubbled event handling | Pattern attaching a single event listener to a parent element to handle events from multiple descendant elements via event bubbling, rather than attaching individual listeners per element. | "Instead of adding a click listener to every item in a list, you add one listener to the list itself and detect which item was clicked — more efficient and works for dynamically added items." *Use `event.target.closest('[data-selector]')` to match delegated targets precisely.* |
| `Intersection Observer` | Scroll detection / viewport detection | Browser API providing asynchronous callbacks when observed elements enter or exit the viewport (or a specified root element), replacing scroll event listeners for visibility detection. | "A browser feature that efficiently watches elements and tells you when they come into or leave the visible area — used for lazy loading, scroll animations, and infinite scroll." *Far more performant than `scroll` event + `getBoundingClientRect()` polling.* |
| `Web Workers` | Background thread | Browser API enabling JavaScript execution in a separate thread from the main UI thread, preventing long-running computations from blocking rendering and user interactions. | "Lets the browser run heavy JavaScript tasks in the background so the page stays responsive and doesn't freeze." *Cannot access the DOM; communicates with the main thread via `postMessage`.* |
| `service worker` | Offline cache / PWA worker | Script running in a background thread separate from the page, acting as a programmable network proxy to intercept requests, manage caches, and enable offline functionality and push notifications. | "A script that sits between your browser and the internet, caching files so the site works offline and loads faster on repeat visits." *Lifecycle: install → activate → fetch intercept. Required for Progressive Web App (PWA) functionality.* |
| `Promise / async-await` | Asynchronous operation | Language constructs for managing asynchronous operations: `Promise` represents an eventual value; `async/await` provides synchronous-style syntax for composing promise chains, with structured error handling via `try/catch`. | "Ways to handle operations that take time — like fetching data — without freezing the page. `async/await` makes the code look like it runs line by line even though it doesn't." *Unhandled promise rejections silently swallow errors — always `catch` or `try/catch` async calls.* |
| `event loop` | JS concurrency model | JavaScript's concurrency mechanism: a single-threaded execution model where the call stack processes one frame at a time, with the event loop continuously moving queued microtasks and macrotasks onto the stack when it empties. | "JavaScript can only do one thing at a time, but the event loop lets it juggle many tasks by queuing them up and executing them in order." *Microtasks (Promise callbacks) always drain before the next macrotask (setTimeout, I/O).* |
| `module system (ESM)` | ES modules / import-export | Native JavaScript module system using `import` / `export` syntax, enabling static analysis, tree shaking, and deferred or dynamic loading via `import()`. | "A standardized way to split JavaScript into separate files that explicitly import and export what they need from each other — keeping code organized and enabling build optimizations." *`type="module"` on `<script>` enables ESM in browsers; always prefer over CommonJS in new code.* |
| `Content Security Policy (CSP)` | XSS protection header | HTTP response header defining trusted sources for scripts, styles, images, and other resources, mitigating cross-site scripting (XSS) and data injection attacks by blocking unauthorized resource origins. | "A security rule that tells the browser which websites are allowed to provide scripts and styles for your page — blocking anything from untrusted sources." *Start with `Content-Security-Policy-Report-Only` to audit violations before enforcing.* |

## Developer Tooling & Workflow

| Preferred Technical Term | Standard / Widely Accepted Term | Description (For Systems) | Guidance (For Humans) |
|---|---|---|---|
| `linter` | Code checker / style enforcer | Static analysis tool that inspects source code for syntactic errors, style violations, anti-patterns, and potential bugs without executing the code. Common tools: ESLint, Stylelint, Biome. | "A tool that reads your code and flags mistakes or bad practices before you run it — like spell-check for code." *Integrate with your editor (via LSP) and CI pipeline for continuous enforcement.* |
| `formatter` | Code prettifier / auto-formatter | Tool that automatically rewrites source code to conform to a defined style specification (indentation, quotes, line length) without changing behavior. Common tools: Prettier, Biome. | "Automatically fixes your code's formatting — spacing, indentation, quotes — so the whole team's code looks consistent without manual effort." *Distinct from a linter: formatters change whitespace/style only; linters find logic and pattern issues.* |
| `CI/CD pipeline` | Automated deployment / build pipeline | Automated workflow (Continuous Integration / Continuous Delivery) that runs tests, linting, builds, and deployment steps on every code change, enforcing quality gates before changes reach production. | "Every time code is pushed, an automated system tests it, builds it, and — if everything passes — deploys it. Catches problems before they reach users." *CI = automated testing on push; CD = automated deployment on passing CI.* |
| `feature flag` | Feature toggle / kill switch | Configuration-driven mechanism enabling or disabling application features at runtime without code deployment, supporting gradual rollouts, A/B testing, and instant rollback. | "A switch in the code that lets you turn a feature on or off for users without redeploying — useful for testing new features with a small group first." *Stored in remote config (LaunchDarkly, Unleash, custom) — not hardcoded conditionals.* |
| `source map` | Debug map | File generated during the build process that maps compiled/minified output code back to the original source files and line numbers, enabling accurate debugging in browser DevTools. | "A file that lets your browser's developer tools show you your original readable code instead of the compressed, unreadable production version when debugging." *Never expose source maps publicly in production — they reveal your source code.* |
| `hot module replacement (HMR)` | Live reload / hot reload | Development server feature that patches updated modules into a running application at runtime without a full page reload, preserving application state between edits. | "When you save a file during development, the change appears in the browser instantly without refreshing — and without losing where you were in the app." *Full HMR (state-preserving) vs live reload (full refresh) — HMR is strictly superior for DX.* |
| `type safety` | TypeScript / static typing | Development practice using a static type system to annotate and enforce the shapes of values throughout a codebase, catching type errors at compile time rather than runtime. | "Adding labels to your data so the computer can check — before running anything — that you're using everything correctly. Catches a whole class of bugs before they happen." *TypeScript compiles to JavaScript; types are erased at runtime — they're purely a development-time tool.* |

## Security

| Preferred Technical Term | Standard / Widely Accepted Term | Description (For Systems) | Guidance (For Humans) |
|---|---|---|---|
| `Cross-Site Scripting (XSS)` | Script injection | Security vulnerability where an attacker injects malicious scripts into content served to other users, executing in their browser context and potentially stealing credentials or session data. | "An attack where someone sneaks harmful code into a website so it runs in other users' browsers — like stealing their login cookies." *Prevented by escaping user-generated content before rendering and enforcing a strict CSP.* |
| `Cross-Site Request Forgery (CSRF)` | Session riding | Attack where a malicious site tricks an authenticated user's browser into making unauthorized requests to another site where the user is logged in, exploiting existing session credentials. | "An attack that tricks your browser into secretly performing actions on a site you're already logged into — like transferring money without your knowledge." *Mitigated via CSRF tokens, `SameSite` cookie attribute, and origin header validation.* |
| `HTTPS / TLS` | SSL / secure connection | Transport Layer Security protocol encrypting data in transit between client and server, ensuring confidentiality, integrity, and server authentication via certificate validation. | "The padlock in your browser — encrypts everything sent between you and the website so nobody in between can read or tamper with it." *HTTP/2 and HTTP/3 require TLS; use HSTS to prevent protocol downgrade attacks.* |
| `same-origin policy` | Cross-origin restriction | Browser security model restricting how documents and scripts from one origin (scheme + host + port) can interact with resources from a different origin, preventing unauthorized cross-site data access. | "Browsers block websites from reading data from other websites unless explicitly permitted — preventing a malicious site from silently reading your bank data." *CORS headers (`Access-Control-Allow-Origin`) are the controlled exception to this policy.* |
| `CORS (Cross-Origin Resource Sharing)` | Cross-origin permission | HTTP header-based mechanism allowing servers to declare which external origins are permitted to access their resources, selectively relaxing the same-origin policy for trusted clients. | "A permission system that lets your website request data from a different website's server — but only if that server explicitly allows it." *Preflight `OPTIONS` request occurs for non-simple requests; server must respond with correct headers.* |
